# Story 9.1: Proc Slot — Require Keybind Before Showing

**Epic:** EPIC-9 — Proc Slot Reliability
**Status:** Ready for Review
**Priority:** High

## User Story

As a Marksman Hunter player, I want the proc alert slot to stay hidden when the glowing spell has no keybind, so I only see actionable procs that I can actually press.

## Background

`UpdateProcAlert` iterates a candidate set built from two sources:
1. `activeOverlaySpells` — populated by `SPELL_ACTIVATION_OVERLAY_GLOW_SHOW` events
2. `cachedRotSpells` / live `C_AssistedCombat.GetRotationSpells()`

For the first candidate that satisfies `spellID ~= primarySpellID` and `C_SpellActivationOverlay.IsSpellOverlayed(spellID)`, the frame is shown. There is **no check** that the spell is on the action bar. On MM Hunter the rotation engine emits overlay events for talent or passive spells not slotted by the player, causing the proc icon to display an unknown spell with no keybind.

`GetSpellKeybind(spellID)` returns `""` when:
- `C_ActionBar.FindSpellActionButtons` finds no real (non-RA) slot for the spell, AND
- `keybindCache[spellID]` has no prior good value.

This is the correct proxy for "not on the action bar". If keybind is empty, the alert is not actionable.

## Acceptance Criteria

- [x] **AC1 — Keybind guard on candidate selection:** In `UpdateProcAlert`, the candidate selection loop (`HekiLight.lua` ~line 1682) must call `GetSpellKeybind(spellID)` for each overlay-active candidate and only accept the candidate if the result is non-empty. If the keybind is empty, continue to the next candidate.
- [x] **AC2 — No show when all candidates lack keybinds:** If every candidate in the set has an empty keybind, `procAlertFrame` must remain hidden (existing hide-path at ~line 1714 applies naturally).
- [x] **AC3 — Correct candidate wins when multiple keybinds exist:** If multiple candidates are overlay-active and have keybinds, the first accepted one is shown (existing behaviour; no priority change required).
- [x] **AC4 — DLog suppression guard respected:** The existing `lastSkippedAlertID` change-detection guard must still prevent log flooding. The new logic does not introduce additional per-frame `DLog` calls.
- [x] **AC5 — showKeybind = false not affected:** The `db.showKeybind` flag controls keybind *text display* only. The new guard operates on `GetSpellKeybind` return value regardless of `db.showKeybind`, so the proc slot still requires a keybind to show even if the text is hidden.
- [ ] **AC6 — In-game verification (MM Hunter):** After the fix, trigger a proc on Marksman Hunter; the proc slot must either show the correct slotted spell or remain hidden. The unknown-spell icon must not appear.

## Implementation Notes

**File:** `HekiLight.lua`
**Function:** `UpdateProcAlert` (~line 1660)

Change the candidate selection loop from:
```lua
for spellID in pairs(candidates) do
    if spellID ~= primarySpellID and C_SpellActivationOverlay.IsSpellOverlayed(spellID) then
        alertSpellID = spellID
        break
    end
end
```
to:
```lua
for spellID in pairs(candidates) do
    if spellID ~= primarySpellID and C_SpellActivationOverlay.IsSpellOverlayed(spellID) then
        if GetSpellKeybind(spellID) ~= "" then
            alertSpellID = spellID
            break
        end
    end
end
```

No other changes needed. The hide-path below the loop handles the "no candidate found" case correctly as-is.
