# Story 5.1: Override-Aware Secondary Slot Deduplication

**Status:** Done
**Epic:** 5 ā€” Correctness Hardening
**Version:** v0.4.0 (TBD)
**Priority:** Implement third ā€” depends on Story 5.3
**Depends on:** Story 5.3 (GetRealSlot for secondary spells)
**Complexity:** S (2-line guard change in existing loop; no structural changes)

## Context

`GetSuggestionQueue` deduplicates secondary spells against the primary with: `if sid ~= primaryID then`. The problem is that `primaryID` is an **override spellID** (returned by `GetNextCastSpell(false)` ā†’ resolved via `GetActionInfo`), while `sid` is a **base spellID** from `GetRotationSpells()`. For specs with talent overrides (e.g., Blood DK with Defile), these IDs are always different ā€” the comparison never fires. The result: the primary spell appears duplicated as the first secondary slot, showing the base icon alongside the override icon.

After Story 5.3, `queueCache` entries carry `overrideSpellID` for secondary slots. This story uses that field to perform a semantically correct dedup comparison.

## Acceptance Criteria

- [ ] The primary spell never appears in the secondary strip, even when it is a talent override
- [ ] Deduplication compares `overrideSpellID` (from Story 5.3) against `primaryID`, not `sid` against `primaryID`
- [ ] When `overrideSpellID` is nil (GetRealSlot returned nothing), fall back to comparing `sid` against `primaryID` ā€” partial protection is better than none
- [ ] Blood DK with Defile active: Defile icon appears once (primary), not twice (primary + secondary)
- [ ] A spec with no talent overrides: behavior is identical to pre-EPIC-5 (regression-free)
- [ ] `DLog("SUGGEST", ...)` entry records when a secondary spell is suppressed by override-aware dedup

## Implementation Notes

- After Story 5.3, each `queueCache[i]` has `overrideSpellID` (the resolved override ID, or `false` if none)
- Change the dedup guard in `GetSuggestionQueue` from:
  `if sid ~= primaryID then`
  to:
  `local effectiveID = entry.overrideSpellID or sid`
  `if effectiveID ~= primaryID then`
- This is a targeted, minimal change to the two-pass fill loop ā€” do not restructure the loop
- The change-detection guard on `DLog("SUGGEST", ...)` should use `effectiveID` as the key, not `sid`, to remain consistent with how the primary is logged

## Scope

**IN:** Override-aware dedup guard in `GetSuggestionQueue` two-pass fill loop; `DLog` on suppressed secondary entry.
**OUT:** No changes to primary spell resolution. No changes to `GetRealSlot`. No structural refactor of `GetSuggestionQueue`.

## Business Value

Directly fixes the #1 visual correctness complaint for talent-override specs: the primary spell appearing twice in the suggestion strip. Restores player trust that the strip shows meaningful upcoming spells, not artifacts.

## Risks

- **Low:** If Story 5.3 ships with a bug where `overrideSpellID` is set to an incorrect value, this story's dedup will suppress the wrong secondary spell. Dependency on 5.3 correctness is therefore critical ā€” do not implement this story until 5.3 is verified in-game with `/hkl log`.
- **Low:** Specs with no talent overrides: `overrideSpellID` will be `false` for all secondary slots; the `or sid` fallback preserves current behavior exactly.
